## 系统总览

本系统采用**四层防御机制**来确保火灾检测的准确性，最大限度减少误报：

```
视频输入 → 第1层: YOLO检测 → 第2层: 10项多模态验证 → 第3层: 三级预警追踪 → 第4层: 连续帧确认 → 🔥 正式报警
```

### 关键设计理念
- **层层过滤**：每一层都有明确的判定标准，未通过则直接拦截
- **双重计数**：采用两个独立计数器分别控制预警升级和最终报警
- **可配置**：所有关键参数支持通过配置文件动态调整

## 第1层：YOLO初步检测

**作用**：快速找出画面中"像火焰"的区域

### 检测流程
1. **输入**：视频每一帧（经过光照增强处理）
2. **模型**：YOLOv8目标检测模型
3. **输出**：带置信度的矩形框 `[xmin, ymin, xmax, ymax]`

### 判定标准
- **置信度阈值**：`confidence ≥ 0.8`（可配置）
- **类别**：仅检测 `fire` 类别（smoke类别已禁用）

### 日志示例
```
YOLO检测到火灾: conf=0.857, box=[1243,721,1527,1057]
```

## 第2层：10项多模态验证

**作用**：用10条规则过滤掉误报（比如灯光、红色物体、照片等）

### 验证内容详解

| 验证项 | 验证内容 | 通过标准 | 失败原因示例 |
|--------|----------|----------|--------------|
| **验证1/10** | 检测框有效性 | 框在画面边界内 | 框超出画面边界 |
| **验证2/10** | ROI面积 | 面积 ≥ 200像素 | 目标太小（<200像素） |
| **验证3/10** | 肤色比例 | 肤色 ≤ 45% | 检测到人的皮肤区域 |
| **验证4/10** | 人脸检测 | 未检测到人脸 | 检测到人脸存在 |
| **验证5/10** | 直线数量 | 直线 < 8条 | 检测到过多直线边缘 |
| **验证6/10** | 火焰颜色比例 | 红色/黄色像素达标 | 火焰颜色像素不足 |
| **验证7/10** | 黄色物体抑制 | 非规则黄色物体 | 像路灯、黄灯等规则物体 |
| **验证8/10** | 高亮区域/轮廓复杂度 | 高亮≤75% 且 复杂度≥1.3 | 太亮或形状太规则 |
| **验证9/10** | 亮度标准差 | V通道标准差达标 | 亮度变化不够（无闪烁） |
| **验证10/10**| 运动比例 | 运动像素达标 | 静止不动的目标 |

### 动态阈值机制
系统根据目标面积自动调整验证阈值：

| 目标规模 | 面积范围 | 火焰颜色比例 | 运动比例 | 亮度标准差 |
|----------|----------|-------------|---------|-----------|
| **small** | < 1500 px | ≥ 0.10 | ≥ 0.08 | ≥ 32.0 |
| **medium** | 1500-6000 px | ≥ 0.06 | ≥ 0.05 | ≥ 25.0 |
| **large** | > 6000 px | ≥ 0.05 | ≥ 0.04 | ≥ 22.0 |

### 日志示例
```
火焰校验通过: scale=large, fire_ratio=0.433, motion=0.004, v_std=84.88
```

## 第3层：三级预警追踪体系

**作用**：连续观察同一个目标，通过追踪确认其稳定性

### 三级预警定义

| 级别 | 颜色 | 名称 | 连续帧数 | 抖动要求 | 含义 |
|------|------|------|---------|---------|------|
| **L1** | 黄色 | Suspected（疑似） | 1帧 | - | 刚发现可疑目标 |
| **L2** | 橙色 | Validating（验证中） | 2-4帧 | - | 看起来像，但还要观察 |
| **L2** | 橙色 | Validating（验证中） | ≥5帧 | jitter < 0.015 | 抖动太小，可能是灯 |
| **L3** | 红色 | Confirmed（已确认） | ≥5帧 | jitter ≥ 0.015 | 确定是火焰！ |

### 两个关键概念

#### 1. Jitter（抖动）
**定义**：目标中心点和面积的变化程度
- **jitter < 0.015**：太稳定 → 可能是灯光、照片 → 维持L2
- **jitter ≥ 0.015**：有跳动 → 符合火焰特征 → 升级到L3

**原理**：火焰会自然跳动，而人工光源是静止的

#### 2. 追踪匹配
系统使用以下三个维度来匹配同一目标：
- **中心点距离**：≤ 80像素
- **IOU（交并比）**：≥ 0.10
- **面积变化**：≤ 2.0倍

### 预警升级逻辑

```python
# 伪代码示意
if frames >= confirm_frames:  # 小目标7帧，大目标5帧
    if jitter < 0.015:
        level = 2  # 抖动太小，维持L2
    else:
        level = 3  # 抖动足够，升级到L3
elif frames >= 2:
    level = 2
else:
    level = 1
```

### 日志示例
```
目标抖动不足，维持在 L2: area=28471, jitter=0.0042, threshold=0.0150
⚠️ 检测到持续 L2 预警目标，继续观察，不触发正式报警
```

### 目标丢失容忍机制（重要！）

**作用**：允许目标短暂丢失（比如某一帧验证失败），不会立即重置追踪状态

#### 核心逻辑（`detector.py` 第537-540行）
```python
for tid in list(unmatched_tracks):
    self.tracked_targets[tid]['misses'] += 1
    if self.tracked_targets[tid]['misses'] > 3:  # 关键：允许连续丢失3次
        del self.tracked_targets[tid]  # 超过3次才真正删除
```

#### 两个阶段分离

| 阶段 | 判定内容 | 结果 |
|------|---------|------|
| **阶段A** | 10项多模态验证 | 决定是否进入 `current_fire_candidates` |
| **阶段B** | 追踪器匹配（考虑misses） | 决定预警级别和追踪状态 |

#### 实际场景示例

假设目标一直存在，但某一帧验证失败（比如直线过多）：

| 检测次 | 10项验证 | `track['frames']` | `track['misses']` | 预警级别 | 说明 |
|-------|---------|-------------------|------------------|---------|------|
| 1 | ✅ 通过 | 5 | 0 | L3 | 正常检测 |
| 2 | ❌ 失败 | 5（保持） | 0→1 | - | 验证失败，但追踪保留 |
| 3 | ✅ 通过 | 5→6 | 1→0 | **L3**（直接！） | 匹配到旧追踪，直接L3，不从L1开始 |
| 4 | ❌ 失败 | 6（保持） | 0→1 | - | |
| 5 | ❌ 失败 | 6（保持） | 1→2 | - | |
| 6 | ❌ 失败 | 6（保持） | 2→3 | - | |
| 7 | ❌ 失败 | - | 3→4 | - | **删除追踪！** misses>3 |
| 8 | ✅ 通过 | 1 | 0 | L1 | **从头开始！** |

#### 关键要点
- ✅ **misses ≤ 3**：追踪器保留，重新匹配后直接回到之前的级别
- ❌ **misses > 3**：追踪器被删除，下次需要从L1重新开始
- 💡 这个机制的意义：避免因偶然验证失败导致整个追踪状态重置

## 第4层：连续帧确认（最终报警）

**作用**：最后一道防线，防止偶然误报

### 两个独立计数器

#### 计数器1：`track['frames']`（追踪器内部）
- **位置**：`detector.py` 第489行
- **作用**：控制三级预警升级
- **范围**：1, 2, 3, ..., N（持续累积）

#### 计数器2：`consecutive_fire_detections`（报警确认）
- **位置**：`main.py` 第199行
- **作用**：控制最终报警触发
- **范围**：0, 1, 2, ..., 6（达到6则报警）
- **重置条件**：目标消失或降级

### 报警触发条件

**必须同时满足**：
1. ✅ 检测到 **L3级别** 的目标
2. ✅ `consecutive_fire_detections` ≥ 6
3. ✅ 距离上次报警 ≥ 360秒（冷却时间）

### 报警动作
1. 保存报警截图到 `output/fire_alert_{timestamp}.jpg`
2. 启动线程发送飞书通知
3. 重置计数器，进入6分钟冷却期

### 日志示例
```
🚨 Detector已确认高级预警 (fire)，进入最终报警确认阶段...
🔥 触发正式报警: 已确认存在 fire!
报警截图已保存: output/fire_alert_1711681723.jpg
```

## 常见问题解答

### Q1: L2预警日志一直出现，为什么不报警？
**A**: 说明系统卡在L2级别，可能原因：
- 目标抖动不足（jitter < 0.015），被判定为可能是灯光
- 目标连续检测帧数未达到5帧
- 目标中途丢失过，导致追踪重置

### Q2: 从L3到报警需要额外6次吗？总共是5+6=11次？
**A**: 不是！总共是10次检测：
- 前5次：从L1→L2→L3（第5次检测时同时达到L3且consecutive_fire_detections=1）
- 后5次：保持L3，让consecutive_fire_detections从1→6
- 第10次检测：触发报警

### Q3: 为什么需要这么多层？直接YOLO检测不行吗？
**A**: 火灾检测容易误报（灯光、红色物体、照片等），多层验证是为了：
- 降低误报率（避免扰民）
- 提高准确率（确保真实火灾不漏报）
- 系统注释提到："Detector内部已经做过连续5帧的严格校验了"

### Q4: 如何让报警更灵敏？
**A**: 可以调整以下参数：
- 降低 `consecutive_threshold`（建议2-3）
- 降低 `detection_interval`（比如改为2）
- 降低 `yolo_confidence`（注意可能增加误报）

### Q5: 抖动不足怎么办？
**A**: 如果真实火焰被误判为抖动不足，可以：
- 降低 `fire_small_target_jitter_min` 阈值
- 或者检查是否摄像头帧率太低导致抖动计算不准确

### Q6: 为什么某帧验证失败后，下一帧通过了直接就是L3，不是从L1重新开始？
**A**: 因为追踪器有**目标丢失容忍机制**（misses机制）：
- 当某帧验证失败时，追踪器不会立即删除，而是 `misses += 1`
- 只有当 `misses > 3`（连续丢失3次）时，追踪器才真正被删除
- 在 `misses ≤ 3` 的情况下，下一帧重新匹配成功会直接使用旧追踪器
- 旧追踪器的 `track['frames']` 已经很高，所以直接就是L3，不从L1开始！

**参考文档**：详见"第3层：三级预警追踪体系"中的"目标丢失容忍机制"章节
